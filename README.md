# Student Planner (Студенческий планер)

## Описание приложения
Приложение «Студенческий планер» предназначено для удобного отслеживания учебных дисциплин, расписания занятий и профиля студента. Реализовано на современном стеке Android с использованием декларативного UI и типизированной навигации. Позволяет быстро просматривать детали предметов, переходить между экранами и управлять настройками приложения.

## Реализованные экраны
- HomeScreen — Главный экран со списком дисциплин
- DetailsScreen — Детальная информация о выбранной дисциплине
- ProfileScreen — Профиль студента с успеваемостью и контактами
- SettingsScreen — Настройки приложения (уведомления, тема, справочная информация)
- ScheduleScreen — Расписание занятий на неделю
- ScheduleDetailScreen — Детальная информация о конкретном занятии

## Используемые технологии
- Kotlin — основной язык разработки
- Jetpack Compose — современный декларативный UI-фреймворк
- Navigation Compose — управление навигацией, back stack и передачей данных между экранами
- Material 3 — компоненты интерфейса по актуальным гайдлайнам Google
- Git/GitHub — контроль версий, коммиты и удалённый репозиторий

## Схема навигации
```
HomeScreen
├──→ DetailsScreen/{subjectId}
├──→ ProfileScreen
├──→ SettingsScreen
└──→ ScheduleScreen
       └──→ ScheduleDetailScreen/{lessonId}
```
(Системная кнопка «Назад» или navigationIcon в TopAppBar возвращают пользователя на предыдущий экран из Back Stack)

## Контрольные вопросы

1. Что такое NavController и для чего он используется? Почему важно создавать его через rememberNavController()?
   NavController — это центральный компонент Navigation Compose, который управляет переходами между экранами, хранит back stack и передаёт аргументы. Функция rememberNavController() сохраняет экземпляр контроллера при рекомпозиции Compose, предотвращая потерю состояния навигации и пересоздание графа маршрутов при каждом обновлении UI.

2. Как передать параметр в маршрут навигации? В чём разница между обязательными и опциональными параметрами?
   Параметр объявляется в маршруте как плейсхолдер: "details/{subjectId}". При переходе вызывается navController.navigate("details/123"). В composable параметр извлекается через backStackEntry.arguments?.getString("subjectId"). Обязательные параметры должны быть указаны всегда, иначе маршрут не сработает. Опциональные задаются через arguments = listOf(navArgument("query") { defaultValue = "" }) и могут отсутствовать в URL.

3. Зачем использовать sealed class для маршрутов?
   sealed class обеспечивает type-safe навигацию. Компилятор проверяет все возможные маршруты, предотвращает опечатки в строковых путях и упрощает рефакторинг. Например, вместо хардкода "details/123" используется Screen.Details.createRoute("123"), что исключает ошибки вида "detail/123" (пропущена буква) на этапе компиляции.

4. Что такое Back Stack и как им управлять? Схема: Home → Profile → Settings. Что произойдёт при popBackStack() на экране Settings?
   Back Stack — это стек посещённых экранов (LIFO). Схема: [Home, Profile, Settings]. При вызове popBackStack() на экране Settings он удаляется из стека, и пользователю автоматически показывается предыдущий экран — Profile. Системная кнопка «Назад» работает по тому же принципу.

5. Как работает startDestination в NavHost? Можно ли изменить его динамически?
   startDestination указывает экран, который будет показан первым при запуске приложения или при очистке back stack. Динамически изменить его в рамках одного NavHost без пересоздания графа нельзя, но можно использовать условную логику перед инициализацией NavHost (например, проверить авторизацию) или настроить deep links для открытия конкретного экрана извне.

6. Что произойдёт, если навигировать на несуществующий маршрут? Как можно обработать такую ситуацию?
   Приложение выбросит IllegalArgumentException: navigation destination ... is not a valid destination. Чтобы обработать ситуацию, можно обернуть вызов navigate() в try-catch, реализовать fallback-экран (например, 404), или валидировать ID перед переходом. В production также используют navController.currentDestination?.route для проверки доступности маршрута.

7. Зачем нужен параметр launchSingleTop в навигации? Приведите пример, когда без него может возникнуть проблема. Как он влияет на back stack?
   launchSingleTop = true предотвращает создание дубликатов экрана в back stack, если пользователь повторно переходит на уже открытый верхний экран. Без этого можно получить стек [Home, Details(1), Details(1)], что ухудшает UX и расходует память. Параметр обновляет существующий экземпляр через onNewIntent, а не добавляет новый в стек.

## Описание реализации Задания 3: Экран «Расписание»
 Ключевые изменения:
- Создана модель данных Lesson, содержащая уникальные идентификаторы, название предмета, день недели, время, аудиторию, ФИО преподавателя и описание.
- Реализован экран ScheduleScreen с выводом занятий, сгруппированных по дням недели. Каждая карточка является кликабельной.
- Разработан экран ScheduleDetailScreen для отображения полной информации о выбранном занятии.
- В TopAppBar главного экрана добавлена кнопка перехода в расписание.
- Настроена маршрутизация в Navigation Compose с обязательным параметром lessonId для безопасной передачи данных.
- Реализована корректная обработка back stack для навигации между экранами расписания и главным экраном.
